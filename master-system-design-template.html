The core idea

Four layers, one request, top to bottom

Almost every backend you will be asked to design does the same four things: a request enters at the edge, gets handled by the application core, sometimes calls a model, and is always observed, evaluated, and secured.

Memorize those four layers once. In the interview you draw them as empty containers, then fill in only the boxes the question needs. A URL shortener barely touches the AI layer. A support copilot lives in it. The frame stays the same; the depth moves to wherever the problem is.

The method

The first five minutes decide the interview

Most candidates lose points before they draw a single box — by jumping to a solution before they know the problem. Spend the opening minutes here, out loud, every time.

  1. Clarify requirements. Separate functional (“users send messages, get replies”) from non-functional (latency, availability, consistency, privacy). Write them where the interviewer can see them.
  2. Estimate scale. Users, requests per second, read/write ratio, payload size, data growth per year. You are not chasing precision — you are choosing between a single database and a sharded fleet.
  3. Draw the four layers as empty boxes. Entry, core, inference, operations. This alone signals structure and buys you thinking time.
  4. Fill only the relevant boxes. Resist drawing everything. Put detail where the requirements pull you, and say why the other layers are thin.
  5. State trade-offs as you go. Every box is a choice. Name the alternative you rejected and the reason. That is the actual thing being graded.
Interviewers are not grading the diagram.

They are grading whether you reason about scale, trade-offs, and failure. A clean diagram with no trade-off talk scores lower than a messy one with sharp reasoning. Narrate your choices.

Layer 1

Entry & edge — get the request in safely

Everything the client touches before your business logic. Its job is to route fast, absorb abuse, and fail cheaply.

01

Client

Web or mobile. Assume it is untrusted, retries on failure, and may disconnect mid-request.

Say: “the client can drop at any point, so long-running work must survive a disconnect.”

02

DNS + CDN

Resolves the domain and serves static assets and cached responses close to the user.

Cuts latency and offloads origin traffic. Mention edge caching and TLS termination.

03

Load balancer

Spreads traffic across service instances, runs health checks, and enables zero-downtime deploys.

Prefer stateless services behind it so any instance can serve any request.

04

API gateway

The single front door: authentication, rate limiting, and routing to the right service.

This is where you protect everything downstream. Rate limit here or the core melts under load.

What to sayAuthentication and rate limiting belong at the edge so a bad actor never reaches the expensive parts of the system.

Layer 2

Application core — do the work, but not on the hot path

The classic workhorse. The single most valuable move in this layer is deciding what happens synchronously versus what gets pushed to the background.

01

Stateless services

Business logic with no local session state, so you can scale horizontally by adding instances.

State lives in the cache and database, never in the service memory.

02

Cache

Redis or similar in front of the database for hot reads.

Name a strategy: read-through, TTLs, and how you invalidate. “Cache invalidation” is a trade-off, not a free win.

03

SQL / NoSQL

Relational for transactions and joins; document or wide-column for scale and flexible schemas.

Justify the choice with the access pattern, then mention indexing, replication, and sharding by a key.

04

Object storage

Blobs, uploads, and large artifacts — images, files, model outputs.

Keep large binaries out of the database; store a pointer instead.

05

Event bus / queue

Decouples slow or spiky work from the request path (Kafka, SQS, etc.).

This is your latency lever. If work can be eventual, it belongs here.

06

Workers

Consume the queue and do the heavy lifting asynchronously — emails, thumbnails, indexing, aggregation.

Make them idempotent so a retried message does not double-charge or double-send.

07

Search + notifications

A search index (Elasticsearch/OpenSearch) and the fan-out that tells users something happened.

Both are fed off the queue, not the request path.

The one lever to rememberWhen latency is a problem, move work off the request path with a queue and let workers finish it later.

Layer 3

AI-native inference — the layer most candidates skip

If the role touches AI, this layer is where you win or lose. The whole thing collapses to one line: Prompt → Context → Model → Tokens.

01

AI gateway / orchestrator

The brain that coordinates the AI request — the equivalent of the API gateway for model calls.

Owns budgets, retries, and the fallback chain when a model is slow or down.

02

Safety + policy

Moderation, PII handling, and compliance before and after the model runs.

Applies to input and output. Say both — a clean prompt can still produce an unsafe answer.

03

Context builder

Assembles the prompt from three sources: RAG / vector DB (retrieved knowledge), memory (past turns), and tools + APIs (live data and actions).

This is where answer quality is actually made or lost — not in the model choice.

04

Model router

Picks the model per request, balancing quality vs cost vs latency — a large model for hard asks, a small fast one for the rest.

Have a fallback matrix by tenant and region for when the first choice fails.

05

Inference server

Where the money burns: batching, KV cache, and GPU utilization (vLLM, TGI, TensorRT-LLM).

Throughput lives here. Mention continuous batching and cache reuse to control cost.

06

Token stream

Streams tokens back as they are generated over SSE or WebSocket.

The metric members feel is time to first token, not total time. Stream, don't wait.

What to sayModel choice is the last decision, not the first. Retrieval, context, and streaming decide whether the answer is good and whether it feels fast.

Layer 4

Data & operations — what makes it production, not a demo

A system that can't be measured can't be trusted. This layer turns raw events into learning, and wraps everything in the four concerns that keep it alive.

The data flywheel

Event stream → lake / warehouse → batch + stream jobs → features + embeddings — and those embeddings feed right back into the RAG store in Layer 3. That loop is how the product gets smarter over time.

Observability

Logs, metrics, traces. One request ID should connect the edge log to the GPU metric to the final response.

AI evals

Quality, safety, and drift. Release gates so a prompt or model change can't silently lower quality.

Security

IAM, secrets, encryption in transit and at rest. Least privilege between every service.

Reliability is the fourth wrap.

Retries with backoff, backpressure when a dependency is slow, and regional failover. Assume every box fails and describe the degraded behavior — that single habit separates senior answers from junior ones.

Reading the diagram

Three line types, three meanings

Half of a good design review is knowing which calls block the user and which don't. Keep the notation honest.

Solid = synchronous

The caller waits for the response. Every solid line on the hot path spends latency budget and adds a way to fail. Keep this chain short.

Dashed = asynchronous

Fire and continue. Work happens later via a queue and workers. This is how you keep the request fast and absorb spikes.

Gold = the AI path

The model flow, called out separately because it has its own latency, cost, and failure profile. It is often async-friendly (stream partial results) even when classic calls are sync.

Rule of thumb

If a dependency doesn't have to finish before the user sees a result, it should be a dashed line.

The checklist

Six questions to answer before you say “done”

Run these at the end of any design. They are the same six checks on the poster, and they map to exactly what interviewers probe.

  • Requirements — did you separate functional from non-functional, and confirm them?
  • Scale + QPS — do your component choices match the numbers you estimated?
  • Latency + SLO — what is the target, and which calls are on the hot path?
  • Data model — does the schema and store fit the access pattern and consistency needs?
  • Failure modes — for each box, what happens when it fails, and what is the degraded path?
  • Cost + trade-offs — what did you optimize for, and what did you consciously give up?

Worked example

“Design a chat assistant for our docs” in four layers

Watch the template adapt. The frame is identical — only the emphasis changes.

L1

Entry & edge

Thin. Client, CDN for the docs site, gateway with auth and per-user rate limits.

Nothing exotic — say so and move on.

L2

Application core

A conversations service, a cache for recent threads, a database for messages, and a queue that kicks off document ingestion when docs change.

Ingestion is background work — dashed line to workers.

L3

AI-native inference

The heart of this one. Context builder pulls from a vector DB of chunked docs, adds conversation memory, routes to a capable model, and streams tokens back.

Go deep here: chunking, retrieval quality, grounding, first-token latency, cost per answer.

L4

Data & operations

Log every question, evaluate answer quality and hallucination rate, and re-embed docs on change to keep retrieval fresh.

Evals are the difference between “it demos” and “we can ship it.”

The pointSame four boxes as a payments system or a URL shortener. You just spent your depth in Layer 3 because that is where this product's hard problems live.

Before you walk in

What separates a strong answer

Do

  • Clarify and estimate before drawing.
  • Draw the four layers first, then fill in.
  • Name a trade-off for every major box.
  • Describe failure and degraded behavior.
  • Stream and cache to protect latency.

Avoid

  • Jumping to a database before the requirements.
  • Drawing every box whether it's needed or not.
  • Putting slow work on the request path.
  • Picking a model before designing retrieval.
  • Ignoring cost, security, and observability.
The template is the floor, not the ceiling.

It guarantees you never freeze and never leave a layer out. Your depth on the one box that matters — and your trade-off reasoning — is what actually earns the offer. And in AI-native rounds, the real skill is judging whether a design (or a model's generated code) will hold up under scale and failure, not just whether it runs.
