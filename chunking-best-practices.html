The Most Important Chunking Diagnostic

When a RAG answer is wrong, do not start by blaming the LLM. First ask whether the chunk containing the answer was actually retrieved and included in the prompt context.

Debug rule: If the gold chunk is absent from Top-K, you have a retrieval or chunking problem. If the gold chunk is present but the answer is still wrong, you likely have a ranking, context placement, prompt, citation, or generation problem.

1. Identify Gold Chunk

Find the source span that contains the correct answer. Record its chunk_id, page, section, and document.

2. Inspect Top-K

3. Classify Failure

Absent means retrieval miss. Present but unused means ranking, prompt placement, or answer synthesis issue.

If the gold chunk is missing

This usually means the chunk was split poorly, embedded without enough context, filtered out, outranked, or never indexed.

  • Check whether metadata filters removed the source document.
  • Try semantic chunking, smaller chunks, or modest overlap.
  • Add title, section, entity, and page context before embedding.
  • Compare vector search, keyword search, and hybrid search.
  • Increase candidate retrieval before reranking.

If the gold chunk is present

The retriever found the evidence, but the final answer path failed to use it correctly.

  • Check if the chunk was ranked too low or placed in the middle of a long prompt.
  • Inspect whether the prompt asks the model to cite sources and abstain when unsupported.
  • Use reranking or reorder the best evidence near the beginning or end of context.
  • Look for contradictory chunks or noisy neighbors.
  • Test whether the answer works when only the gold chunk is provided.

Top Interview Questions

Strong RAG candidates do not just say "use chunk size 500." They explain tradeoffs, evaluation, failure modes, and how they would prove a chunking strategy works.

How do you know whether the right chunk was included in context?

Create a golden eval set where each question maps to one or more source chunks. For every query, log retrieved chunk IDs, rank, score, reranker score, and final prompt context. Then compute hit rate, recall@k, and MRR against the gold chunks.

In production, traces should show the exact chunks sent to the model. If the source span is not in Top-K or not in the final prompt, the LLM cannot cite it reliably.

What chunk size would you choose?

Start from the document type and query type, not a universal number. Common text ranges are roughly a few hundred tokens per chunk, but dense policies, APIs, tables, code, transcripts, and FAQs need different boundaries.

Smaller chunks can improve precision but lose context. Larger chunks preserve context but may dilute embeddings and add irrelevant text. The real answer is to test multiple sizes against retrieval metrics and answer quality.

Fixed-size chunking or semantic chunking?

Fixed-size chunking is simple and predictable. Semantic chunking respects headings, paragraphs, sections, tables, and code blocks, which often makes chunks more self-contained.

A strong default is structure-aware recursive splitting: split by document structure first, then paragraph/sentence boundaries, then token limits as a fallback.

How much overlap should chunks have?

Use overlap to protect answer spans that cross boundaries. Too little overlap drops context. Too much overlap creates duplicate retrieval results and wastes context budget.

Start with modest overlap, then tune using recall@k, duplicate rate, and answer faithfulness. If overlap is doing too much work, your boundaries may be wrong.

What metadata should every chunk store?

At minimum: chunk_id, document_id, source URL or file, page number, section heading, token range, timestamp or version, access-control tags, and parent document reference.

Metadata helps filtering, citations, debugging, re-indexing, and permission checks. Missing metadata turns retrieval debugging into archaeology.

What is contextual retrieval?

Contextual retrieval adds identifying document-level context to each chunk before embedding or indexing. For example, a chunk from a revenue table may get a prefix saying which company, quarter, report, and section it came from.

This helps when a chunk is locally ambiguous. "Revenue grew 3%" is weak alone; "ACME 2023 Q2 10-Q revenue grew 3%" is much easier to retrieve for the right query.

How do you handle tables, code, and diagrams?

Do not blindly split them like prose. Tables often need row groups, column headers, captions, and page context. Code needs function/class boundaries. Diagrams may need extracted descriptions plus the surrounding text.

The retrieval unit should preserve the meaning a human would need to answer a question.

What is parent-child chunking?

Use small child chunks for retrieval, then return a larger parent section to the LLM. This can combine precise search with enough surrounding context for answer generation.

It works well when exact sentences retrieve best, but the model needs the paragraph, section, table caption, or policy exception around them.

Why can adding more chunks make answers worse?

More context can add hard negatives, contradictions, duplicated text, and irrelevant passages. Long-context models can also underuse evidence placed in the middle of a long prompt.

Increasing k is not a free improvement. Measure whether additional chunks raise recall without hurting precision, citation quality, latency, and token cost.

How would you evaluate a chunking strategy?

Build an eval set of realistic questions with gold source spans. Measure retrieval quality before generation, then measure answer quality with citations after generation.

Compare chunk sizes, overlap, semantic boundaries, metadata enrichment, hybrid search, and reranking. Keep the test fixed so improvements are real and repeatable.

How do permissions affect chunking?

Every chunk must carry access-control metadata. Retrieval should filter unauthorized chunks before they enter the prompt, not after answer generation.

A chunking strategy that ignores permissions can create data leaks even when the model itself behaves correctly.

What signals show chunking is bad?

The right document appears but the answer span does not. Retrieved chunks start or end mid-thought. Citations point to broad irrelevant passages. Similar chunks dominate the prompt. Users complain that answers miss exceptions, definitions, or qualifiers.

The fix is usually better boundaries, metadata, contextual prefixes, reranking, or parent-child retrieval, not simply a larger model.

Best Practices

1. Design chunks around questions

Chunking is not just document preprocessing. It is an information access design problem. Start by listing the questions users ask, the evidence needed, and the natural boundaries in the source material.

2. Preserve semantic boundaries

Prefer headings, paragraphs, list items, table regions, code blocks, and transcript turns over arbitrary character counts. Use token limits as constraints, not as the only logic.

3. Make chunks self-contained

A chunk should include enough local context to stand alone in search results. Add section titles, document title, entity names, page number, and date when the text would otherwise be ambiguous.

4. Keep chunk IDs stable

Stable IDs make evals, citations, cache invalidation, and debugging possible. If content changes, version the chunk instead of silently replacing evidence under the same ID.

5. Tune overlap with evidence

Overlap should protect boundary-spanning answers. It should not create a wall of near-duplicate chunks. Track duplicate retrieval rate and recall improvements together.

6. Retrieve more, then rerank

A common production pattern is to retrieve a larger candidate set, then use a reranker to choose the final context. This separates broad recall from final precision.

7. Use hybrid search when language varies

Dense vectors help semantic similarity. Keyword or BM25-style search helps exact terms, product names, IDs, policy codes, and error messages. Hybrid retrieval often beats either one alone for enterprise documents.

8. Keep citations tied to chunks

The final answer should cite the chunks it used. If a model gives an answer without citing the gold chunk, your system should treat that as a reliability signal.

9. Do not overstuff context

More chunks can increase noise. Prefer fewer high-quality chunks with clear provenance over many loosely related passages. Reorder important evidence intentionally.

10. Re-index deliberately

Chunking changes should trigger evals and controlled re-indexing. Store the chunking config with the index so you can explain why a chunk exists and reproduce results.

Metrics That Matter

Separate retrieval metrics from answer-generation metrics. If retrieval is broken, answer quality metrics alone will hide the root cause.

Metric What It Answers Why It Matters
Hit Rate@K Did at least one gold chunk appear in the top K? Fast signal for whether retrieval found the needed evidence at all.
Recall@K What fraction of relevant chunks were retrieved? Useful when questions need multiple chunks, such as comparison or synthesis tasks.
MRR How high did the first relevant chunk rank? High rank matters because final context windows are limited and rerankers may trim results.
NDCG Are better chunks ranked above weaker chunks? Helps evaluate graded relevance, not just binary relevant/not relevant.
Citation Accuracy Does the answer cite the chunks that support it? Separates grounded answers from plausible but unsupported generations.
Duplicate Rate Are near-identical chunks crowding out diversity? High overlap or repeated boilerplate can waste Top-K slots.
Abstention Quality Does the system say "not enough evidence" when chunks are missing? A reliable RAG system should fail closed, not invent missing facts.

Debugging Playbook

  1. Start with a failed user question. Capture the query, answer, citations, retrieval logs, reranker logs, final prompt, and model output.
  2. Find the gold source span. Identify the document, page, section, and exact text needed to answer correctly.
  3. Map source span to chunk IDs. If the source span was split across chunks, flag a boundary or overlap issue.
  4. Inspect Top-K retrieval. If the gold chunk is absent, compare vector, keyword, and hybrid retrieval. Check filters and permissions.
  5. Inspect final prompt context. If the gold chunk was retrieved but trimmed before generation, adjust reranking, context budget, or packing logic.
  6. Test the model with only the gold chunk. If it still fails, improve the prompt or answer format. If it succeeds, retrieval/context assembly is the problem.
  7. Run the eval suite. Confirm that the fix improves retrieval and answer quality without regressing other query types.

Production Checklist

Indexing

  • Chunk config is versioned and reproducible.
  • Chunk IDs are stable and traceable to source spans.
  • Document title, section, page, and ACL metadata are attached.
  • Tables, code, and lists use custom splitting rules.

Retrieval

  • Top-K logs include IDs, ranks, scores, and filters.
  • Hybrid search and reranking are evaluated against baselines.
  • Duplicate chunks are detected and controlled.
  • Candidate retrieval is separated from final context packing.

Generation

  • Prompt includes citations and abstention instructions.
  • Important evidence is not buried in noisy middle context.
  • Answers cite source chunks, not just documents.
  • Contradictory chunks are handled explicitly.

Evaluation

  • Golden questions map to source spans or chunk IDs.
  • Retrieval metrics are tracked separately from answer metrics.
  • Tests cover short facts, multi-hop questions, tables, and edge cases.
  • Chunking changes require eval comparison before rollout.
