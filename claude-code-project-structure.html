The core idea

Stop hoping the model behaves. Engineer how it behaves.

A giant instruction file turns every task into a context dump. A structured Claude Code repo turns behavior into a system: stable rules, reusable commands, specialized context, and automated checks.

That matters because the best AI engineers are not only good at prompting. They are good at shaping the environment around the model, then reading and verifying the output with a clear standard.

The folders

What each piece is responsible for

The clean split is simple: one file says what the project is; the folders say how the agent should operate inside it.

01

CLAUDE.md

The project overview and default instructions.

Keep it stable: mission, stack, architecture notes, testing commands, style expectations, and anything Claude should know before touching code.

02

.claude/

The hidden operating system for the repo.

This is where behavior becomes modular. Instead of stuffing every instruction into one file, you separate rules, commands, skills, agents, and hooks.

03

rules/

Coding conventions the agent can actually follow.

Use rules for durable standards: how errors are handled, how components are styled, what tests must accompany a change, and what patterns are off-limits.

04

commands/

Your repeatable slash workflows.

A good command turns a fuzzy request into a known sequence: inspect, edit, test, summarize. It saves the team from re-explaining the same workflow every time.

05

skills/

Context that loads only when relevant.

This keeps the model sharper. A database task gets database context; a design task gets design context. The prompt stays lighter because the repo knows when to load the details.

06

agents/

Specialized workers with their own focus.

Use agents when a task benefits from a narrow lens: reviewer, tester, migration planner, documentation pass, security check, or performance investigation.

07

hooks/

Event-driven scripts that protect the repo.

Hooks are where trust becomes verification: block risky shell commands, require formatting, run targeted tests, or warn before a destructive change gets through.

AI-engineer angle

The repo is part of the prompt

When an AI assistant produces bad code, the failure is not always in the model. Sometimes the repo never told it what good looks like.

Read the structure

Before trusting output, check which instructions, skills, commands, and guardrails were available to the agent.

Verify the workflow

Make sure the model inspected the right files, used the right conventions, and ran the right tests for the change.

Constrain the blast radius

Use hooks and clear rules for risky operations so the assistant cannot silently wander outside the task.

The shift is subtle but big.

You move from "I can prompt it" to "I can direct it." That is the same instinct you use in code review: define the standard, inspect the diff, and ask what would break.
