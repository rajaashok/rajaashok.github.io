The core idea

Choose the architecture before choosing the framework.

Agent projects become fragile when implementation starts with a model name or orchestration library. This map starts with outcomes, decomposes the system into ten durable concerns, and only then places tools against those decisions.

The animation uses a breadth-first reveal for the same reason. First see the whole system. Then understand the responsibilities inside every layer. Only after the contracts are visible should products and frameworks enter the conversation.

The ten-layer system

What each branch is responsible for

Treat each row as a design review. A production agent is ready only when the team can answer the questions in every layer.

01

Strategy and planning

Define why the agent exists before deciding how it works.

  • Write concrete use cases, user personas, success metrics, and constraints.
  • Separate tasks that need agency from tasks better served by deterministic software.
  • Use Notion, Miro, Airtable, Figma, or Jira only to make decisions visible.
02

Agent design

Turn the outcome into a bounded workflow.

  • Choose the agent architecture and decompose the work into observable tasks.
  • Define workflow transitions, tool contracts, stop conditions, and failure paths.
  • Diagram the logic before code with Excalidraw, Lucidchart, Whimsical, or diagrams.net.
03

Model layer

Route work to the smallest model that can do it reliably.

  • Evaluate model quality, latency, context limits, cost, and data policy.
  • Version prompts, validate structured outputs, and add guarded fallbacks.
  • Keep provider boundaries replaceable across proprietary and open models.
04

Memory and knowledge

Store only what improves the next decision.

  • Separate short-term working state from durable user or business memory.
  • Choose vector storage and knowledge sources according to retrieval needs.
  • Design the RAG pipeline for freshness, permissions, citations, and evaluation.
05

Tools and integrations

Give the model narrow, typed, auditable actions.

  • Wrap APIs and services in explicit function schemas with authentication.
  • Define idempotency, timeouts, retries, permission boundaries, and side effects.
  • Use connectors and plugins as adapters, not as hidden business logic.
06

Orchestration and logic

Make state transitions durable and inspectable.

  • Choose a framework only after the state machine and recovery model are clear.
  • Use workflow orchestration for checkpoints, retries, schedules, and long-running work.
  • Add multiple agents only when role separation beats one well-designed workflow.
07

Guardrails and safety

Control what enters, what leaves, and what requires review.

  • Validate input, output, tool arguments, policies, and business rules.
  • Escalate high-impact, low-confidence, or irreversible actions to a human.
  • Evaluate safety behavior continuously rather than relying on one prompt.
08

Observability and debugging

Capture enough evidence to explain every run.

  • Trace model calls, prompts, retrieval, tool use, state transitions, and costs.
  • Log structured events with correlation IDs and privacy-aware payloads.
  • Test workflows against representative tasks and known failure cases.
09

Deployment and infrastructure

Ship an agent as a governed service, not a notebook.

  • Select cloud, container, and scaling patterns from workload behavior.
  • Automate builds and deployments while keeping secrets out of code and traces.
  • Plan queues, concurrency, rate limits, budgets, rollback, and regional constraints.
10

Monitor and improve

Close the loop between production evidence and product decisions.

  • Monitor task success, quality, latency, cost, safety, and user correction rates.
  • Segment analytics by workflow, user intent, model, and release version.
  • Convert feedback into datasets, evaluations, experiments, and targeted fixes.

How to use the map

Three passes for a real design review

Pass 1: coverage

Walk the ten branches and mark each one explicit, assumed, missing, or intentionally deferred. Missing operations and safety decisions usually surface here.

Pass 2: contracts

For every edge, name the input, output, owner, timeout, failure behavior, and evidence emitted. This turns a pretty diagram into an executable design.

Pass 3: proof

Attach a test, metric, trace, or review gate to every important claim. If the team cannot observe it, the system cannot reliably improve it.

The AI-engineer review question

When a coding model generates an agent, can you verify the state model, tool permissions, recovery behavior, evaluation plan, and operational evidence, or are you trusting the happy path because the demo looks convincing?

Production checkpoint

Questions to answer before launch

Behavior and safety

  • What can the agent decide, recommend, and execute?
  • Which actions require confirmation or human approval?
  • What happens when a model, tool, or knowledge source fails?
  • How are prompt injection and data access boundaries tested?
  • Can every material action be reconstructed from traces?

Quality and operations

  • Is there a versioned evaluation dataset for core tasks?
  • Are latency, token cost, tool failures, and task success measured together?
  • Can the workflow resume safely after a crash or timeout?
  • Are provider, model, and framework choices replaceable?
  • Does production feedback produce the next evaluation case?
