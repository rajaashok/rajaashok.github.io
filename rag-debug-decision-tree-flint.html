How to use this tree

Debug RAG from evidence upward.

A wrong answer is rarely just a “bad model” problem. It is usually a broken chain: source → parse → chunk → embed → index → retrieve → inject → generate.

Rule 01

Never start with the LLM

First prove whether the answerable fact exists in the source corpus. If the source is missing, stale, duplicated, or contradictory, no retrieval strategy can rescue the answer reliably.

Rule 02

Inspect the actual chunks

Do not debug from scores alone. Read the top retrieved chunks exactly as the model saw them. Most RAG bugs become obvious once you inspect the evidence packet.

Rule 03

Separate retrieval from generation

If the correct chunks were not retrieved, debug indexing and search. If the correct chunks were retrieved, debug context injection, prompt instructions, citations, and model grounding.

The diagnostic path

Start at the top of the tree and follow the first question you can answer with evidence. Every branch should be backed by logs, source snippets, retrieved chunk text, prompt payloads, or index metadata.

Fast workflow

  1. Pick one failing user question.
  2. Write the expected answer in one sentence.
  3. Find the exact source document that contains the answer.
  4. Inspect the retrieved chunks for that query.
  5. Inspect the final prompt sent to the model.
  6. Fix the first broken link in the chain.

Golden rule

Do not ask, “Why did the model hallucinate?” first. Ask, “What evidence did the model receive, and was that evidence complete, fresh, and unambiguous?”

Once you answer that, the next move is usually mechanical: refresh docs, fix parsing, adjust chunking, rebuild embeddings, repair index versioning, tighten filters, rerank, or improve prompt grounding.

Branch-by-branch documentation

Tree node What to inspect Typical fix
Correct info in source docs? Open the canonical source. Check version, tenant, access policy, timestamp, duplicate copies, and contradictory documents. Refresh or remove documents, resolve conflicts, and add a known-answer regression question.
Correct chunks retrieved? Log top-k chunks with document id, chunk id, score, rank, metadata filters, and exact text shown to the model. Fix retrieval before touching the prompt. Tune query rewriting, filters, hybrid search, reranking, or k.
Documents parsed correctly? Compare original PDFs, HTML, tables, headings, images, and extracted text. Look for missing tables, broken order, OCR errors, or boilerplate. Improve extraction, OCR, table parsing, layout parsing, and source-specific cleanup.
Chunks too big / small / split? Check whether the answer spans multiple chunks, whether headings were detached, and whether overlap preserves the needed context. Change chunk size, overlap, semantic boundaries, parent-child retrieval, or contextual chunk headers.
Embeddings regenerated? Confirm every changed chunk has a fresh embedding using the current embedding model and schema version. Rebuild embeddings and tag them with model name, dimension, source version, and build timestamp.
Vector index updated? Check whether the deployed index contains the new embeddings and whether deleted chunks are actually removed. Rebuild or swap the index atomically. Avoid mixing old and new index versions.
Search / filters / rank Inspect metadata filters, tenant filters, ACL filters, lexical search, vector search, hybrid weights, and reranker inputs. Fix filter logic, add metadata tests, tune ranking, add reranking, or rewrite the query.
Prompt included context? Inspect the final prompt payload, not the template. Verify chunk text, citations, ordering, truncation, and system instructions. Fix context injection, token budgeting, citation formatting, and prompt assembly.
Model followed context? Compare answer claims to provided chunks. Look for unsupported claims, ignored citations, or conflicts inside the context. Improve grounding instructions, require citations, lower creativity, change model, or resolve conflicting docs.

After every document refresh

Ingestion
  • Confirm the new document uploaded.
  • Confirm parsing produced complete text.
  • Confirm old versions were removed or marked inactive.
Indexing
  • Regenerate chunks and embeddings.
  • Update the vector index atomically.
  • Verify metadata, tenant, and access filters.
Evaluation
  • Run known test questions.
  • Inspect top retrieved chunks.
  • Verify the answer cites the right evidence.
