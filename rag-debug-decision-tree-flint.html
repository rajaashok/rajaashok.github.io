RAG operations

RAG debugging decision tree

Start with the evidence the model received. Then trace the system downward: source, parse, chunk, embed, index, retrieve, inject, generate.

top-down flow
retrieval first
generation second

Field guide

Use the tree as a fault isolation path.

Pick one failing question. Move through the tree only when you can prove the previous step with an artifact: source text, parse output, chunk text, index metadata, retrieved results, or the final prompt payload.

Principle 01

Verify the source first

If the answer is missing, stale, duplicated, or contradicted in the corpus, retrieval and prompting will only hide the real problem.

Principle 02

Read the retrieved chunks

Scores are not enough. Inspect the exact text, order, metadata, and citations the model received.

Principle 03

Separate retrieval from generation

Missing evidence is a retrieval problem. Ignored evidence is a prompt, context, model, or conflict problem.

Fast debugging loop

Keep the loop small. One question, one expected answer, one source of truth, one trace through the system.

Runbook

  1. Choose a failing user question.
  2. Write the expected answer in one sentence.
  3. Open the canonical source document.
  4. Inspect the parse output and chunk boundaries.
  5. Log the top retrieved chunks with ids and scores.
  6. Inspect the final prompt sent to the model.
  7. Fix the first broken link and add a regression question.

Decision rule

Do not begin with "the model hallucinated." Begin with: what evidence did the model receive, and was that evidence complete, current, scoped to the user, and unambiguous?

Once that is clear, the fix is usually mechanical: refresh documents, repair parsing, adjust chunking, rebuild embeddings, swap the index, correct filters, rerank, or tighten prompt grounding.

Branch-by-branch guide

The table maps each decision point to the artifact worth inspecting and the likely repair.

Decision point Inspect Likely repair
Correct info in source docs? Canonical document, version, timestamp, tenant, access policy, duplicates, and conflicts. Refresh or remove documents, resolve conflicts, and add a known-answer test.
Correct chunks retrieved? Top-k chunk ids, document ids, score, rank, metadata filters, and exact text. Tune retrieval before prompting: query rewrite, filters, hybrid search, reranking, or k.
Documents parsed correctly? Original PDF or HTML beside extracted text. Check tables, headings, OCR, order, and boilerplate. Improve extraction, OCR, table parsing, layout parsing, or source-specific cleanup.
Chunks too big, small, or split? Whether the answer spans chunks, loses headings, or needs adjacent context. Change chunk size, overlap, semantic boundaries, parent-child retrieval, or contextual headers.
Embeddings regenerated? Embedding model, dimension, chunk hash, source version, and build timestamp. Rebuild embeddings for every changed chunk and version the embedding build.
Vector index updated? Deployed index version, deleted chunk tombstones, and index build logs. Rebuild or atomically swap the index. Avoid mixing old and new versions.
Search, filters, rank Tenant filters, ACL filters, metadata predicates, lexical results, vector results, and reranker input. Fix filter logic, add metadata tests, tune weights, rerank, or rewrite the query.
Prompt included context? Final prompt payload, not just the template. Check truncation, order, citations, and instructions. Repair context injection, token budgeting, citation formatting, and prompt assembly.
Model followed context? Answer claims against supplied chunks. Look for unsupported claims, ignored citations, and conflicts. Tighten grounding instructions, require citations, reduce creativity, change model, or resolve source conflicts.

Refresh checklist

Run this after every corpus refresh, parser change, chunking change, or embedding model change.

Ingestion
  • New documents uploaded.
  • Parsing produced complete text.
  • Old versions removed or marked inactive.
Indexing
  • Chunks and embeddings regenerated.
  • Vector index updated atomically.
  • Metadata, tenant, and ACL filters verified.
Evaluation
  • Known questions re-run.
  • Top chunks inspected manually.
  • Answers cite the expected evidence.
