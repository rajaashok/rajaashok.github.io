The LLM Cost Tree: 15 Levers to Lower Cost Without Lowering Quality

The useful optimization target is not the cheapest token. It is the lowest cost for a task that actually succeeds.

July 2026 LLM economics Production guide 18 min read

Cost is a system, not a price list

A cheaper model can become more expensive when it retries, calls too many tools, produces unusable output, or needs a frontier model to repair the result. A more expensive model can be economical when it succeeds in one bounded pass.

The tree organizes the optimization problem into five surfaces: spend less per call, send fewer tokens, generate less, avoid repeated work, and control execution. The article below turns every leaf into an implementation decision and a test.

COST PER SUCCESS

The operating metric

Measure cost per successful task

Token spend is an input metric. Successful work is the outcome.

If one workflow costs $0.03 per run but succeeds only half the time, its effective model cost is already $0.06 per successful result before support, review, and reprocessing. If a $0.05 workflow succeeds 95% of the time, it may be the better production choice.

This is why naive cost optimization is dangerous. Reducing context may lower input tokens while increasing hallucinations. Switching every request to a small model may lower the invoice per call while increasing retries. Aggressive semantic caching may create cheap answers that are stale or wrong.

The denominator forces discipline. Define success for the task, record the cost of every attempt, and optimize the full path until the successful outcome is cheaper.

The map

Five branches, fifteen levers

Use the tree as a diagnostic sequence. Start with the branch that matches the waste you can observe.

ASpend less per call BSend fewer tokens CGenerate less DAvoid repeated work EControl execution

The order matters: remove waste before negotiating price. A discount on unnecessary work is still unnecessary work.

A

Model choice

Spend less per call

Match model capability to the task, then pay for escalation only when the evidence says you need it.

01

Right-size the model

Simple task → smaller model

Classify the work before selecting the model. Extraction, tagging, normalization, formatting, and narrow retrieval-grounded answers often need less reasoning than architecture review, ambiguous debugging, or multi-document synthesis.

Do not decide from model reputation. Build a representative evaluation set and compare success rate, latency, and total cost. The smallest model that clears the quality threshold is the right model for that task class.

First test: replay 100 real tasks across two model tiers and compare cost per accepted result.

02

Route by complexity

Classify → small or frontier

A router can use task type, input length, domain, requested action, risk level, or a lightweight classifier. The router itself should be cheap, deterministic where possible, and observable.

Keep the policy understandable. If every routing decision depends on another unconstrained LLM call, you may simply move the cost. Start with explicit rules, then add learned routing only when the data shows a meaningful gain.

First test: log the router decision and calculate success rate for each route, including misroutes.

03

Escalate selectively

Cheap first → frontier if needed

Escalation works when failure is detectable. Trigger it on schema failure, low retrieval confidence, missing citations, failed verification, conflicting evidence, policy-sensitive intent, or an explicit uncertainty signal backed by tests.

Avoid blind double inference. If every small-model response is automatically reviewed by a larger model, the system has not saved a call. Escalate only when a measurable gate fails.

First test: measure how often escalation changes an incorrect result into a correct one.

B

Context

Send fewer tokens

The model should see the evidence required for the current decision, not the entire history of the system.

04

Trim context

20K history → 2K relevant

Remove duplicated instructions, stale turns, boilerplate, navigation text, irrelevant tool output, and documents that cannot affect the answer. Preserve the current task, active constraints, evidence, and decisions that still matter.

Token trimming needs regression tests. A smaller prompt is only an improvement when task success holds. Track which removed context later causes repair calls; that is evidence your trimming policy is too aggressive.

First test: compare full-history and trimmed-history outputs on long conversations with known answers.

05

Summarize memory

Long thread → compact state

Replace old conversational detail with durable state: goals, decisions, constraints, unresolved questions, entities, and source references. Keep the recent turns verbatim so the model can interpret the current exchange.

Summaries are lossy. Store provenance or links to the original events, version the summary, and regenerate it when contradictions appear. Never let a summary silently become the only copy of a critical fact.

First test: ask factual questions from early in the thread after multiple summary updates.

06

Retrieve, do not stuff

Query → top-k evidence

Retrieve the smallest set of passages that can support the answer. Use metadata filters, hybrid search, reranking, and task-aware chunking so relevant evidence appears before the context budget is exhausted.

Do not optimize only for fewer chunks. Measure retrieval recall on a gold set. Missing the one decisive passage can create a cheap prompt and an expensive failure.

First test: measure recall@k and answer success as k changes.

C

Output

Generate less

Output tokens are expensive, but the larger waste is output that downstream code cannot use.

07

Cap output

Schema + token ceiling

Set a maximum output budget that matches the product surface. A three-field classifier does not need an essay. A support response may need a few paragraphs. A code generator needs room for a complete artifact, but not unlimited narration.

Combine token ceilings with explicit length, field, and completion requirements. Watch for truncation: a low cap that produces incomplete JSON or half-written code simply converts token savings into retries.

First test: chart output length, truncation rate, and accepted-result rate by task type.

08

Use structured output

Valid JSON → fewer retries

When software consumes the response, define the schema at the model boundary. Typed fields remove parsing ambiguity, make validation deterministic, and let the application distinguish a refusal or missing value from malformed prose.

A schema is not semantic verification. Validate ranges, identifiers, permissions, and business rules after parsing. Structured nonsense is still nonsense.

First test: compare parser failures and repair calls before and after strict schema enforcement.

09

Tool before model

Lookup first; reason second

Use deterministic systems for arithmetic, database lookups, search, filtering, date calculations, policy retrieval, and state changes. Give the model the result and ask it to reason or communicate.

This reduces hallucination and often shrinks the prompt. It also creates an auditable boundary: the database supplied the balance; the model explained it. The model did not invent the balance.

First test: list every fact the model currently computes that an authoritative tool could return.

D

Reuse

Avoid repeated work

If the inputs and acceptable outcome have not changed, do not buy the same reasoning again.

10

Cache exact responses

Same input → instant hit

Exact caching is the safest reuse strategy. Build the key from normalized input, model, prompt version, tool version, relevant permissions, and data freshness. Include every field that can change the correct answer.

Use explicit expiration and invalidation. A cache without ownership becomes a silent source of stale answers.

First test: calculate hit rate and inspect a sample of hits for false equivalence.

11

Cache prompt prefixes

Reuse stable instructions

Put stable system instructions, tool definitions, policies, and long reference material before the changing user content. Provider prompt caching can reuse this prefix when requests share the same beginning.

Keep the prefix genuinely stable. Timestamps, random IDs, reordered tools, or changing metadata near the front can break cache reuse. Track cached-token usage instead of assuming the cache is working.

First test: inspect provider usage fields and verify cache hits after the first warm request.

12

Cache semantically

Similar intent → reuse

Semantic caching can reuse an answer for meaningfully equivalent inputs, but it changes the correctness model. Use it for tolerant, low-risk, slowly changing domains. Avoid it when identity, current state, exact numbers, or subtle wording changes the answer.

Store the matched query, similarity score, answer provenance, model version, and freshness window. Apply domain and permission filters before similarity search.

First test: review false-positive hits near the similarity threshold before enabling automatic reuse.

E

Control

Control execution

Agent cost is not one model call. It is every turn, tool call, retry, wait, and repair inside the run.

13

Batch async work

Many jobs → one batch

Move non-interactive work such as nightly classification, embedding, evaluation, summarization, and document processing out of the request path. Queue it, deduplicate it, and use provider batch facilities where the latency window fits.

Batching also improves operational control: you can cap concurrency, group similar workloads, retry failed records, and reconcile outputs by request ID.

First test: separate workloads by latency requirement and identify everything that does not need a live response.

14

Bound agent loops

Turns, tools, timeout

Set hard limits for turns, tool calls, wall-clock time, token usage, and spend. Add idempotency keys for writes. Detect repeated tool calls with unchanged arguments. Stop when progress stalls or the same verification fails repeatedly.

The stop condition should produce a useful state: partial result, reason for stopping, work completed, evidence collected, and what a human should do next.

First test: run adversarial tasks that cannot be completed and verify the agent stops predictably.

15

Measure the outcome

Tokens, latency, success

Record cost and usage per run, but join them to the task result. Track accepted answer, completion status, verification result, human override, retry count, latency, model route, cache status, and tool activity.

Aggregate by task class and prompt version. Averages hide expensive tails, so inspect p95 cost, p95 latency, and the small number of runs that consume a disproportionate share of the budget.

First test: produce a weekly table of cost per success by workflow, route, and failure reason.

Reference implementation

Put the budget in code

A production request should carry explicit limits and produce an outcome record. The example is provider-neutral on purpose.

pythonroute, budget, verify, measure
from dataclasses import dataclass
from time import monotonic

@dataclass(frozen=True)
class RunBudget:
    max_input_tokens: int = 8_000
    max_output_tokens: int = 800
    max_turns: int = 6
    max_tool_calls: int = 10
    timeout_seconds: int = 45
    max_cost_usd: float = 0.25

def choose_model(task):
    if task.risk == "high" or task.requires_deep_reasoning:
        return "frontier"
    return "small"

def run_task(task, budget=RunBudget()):
    started = monotonic()
    model = choose_model(task)
    context = retrieve_relevant_context(task, limit=6)
    cache_key = exact_cache_key(task, model, context.version)

    if cached := exact_cache.get(cache_key):
        return record_outcome(task, cached, cache_hit=True)

    state = AgentState()
    while state.turns < budget.max_turns:
        enforce_budget(state, budget, started)
        response = call_model(
            model=model,
            task=task,
            context=context,
            state=state.compact_memory(),
            output_schema=ResultSchema,
            max_output_tokens=budget.max_output_tokens,
        )

        if response.tool_request:
            result = execute_approved_tool(response.tool_request)
            state.add_tool_result(result)
            continue

        if verify(response, task):
            exact_cache.put(cache_key, response)
            return record_outcome(task, response, success=True)

        if model == "small" and should_escalate(response, state):
            model = "frontier"
            state.note("escalated_after_failed_verification")
            continue

        state.note("verification_failed")

    return record_outcome(task, state.partial_result, success=False)

This shape makes the economics reviewable. Routing is explicit. Context has a limit. Caching is versioned. Tool calls are controlled. Output follows a schema. Escalation follows a failed gate. The loop has a stop condition. Every exit records an outcome.

The fastest optimization is usually the one that explains a visible failure mode.

Rollout order

A practical sequence for an existing product

  1. Instrument before changing behavior. Record model, prompt version, tokens, latency, retries, tools, cache state, cost, and success for every run.
  2. Remove obvious waste. Delete duplicated prompt text, cap unreasonable outputs, stop repeated tool calls, and move non-live work to queues.
  3. Create task-level evaluations. Cost optimization without a stable quality test is guesswork. Use production examples and real acceptance criteria.
  4. Introduce routing and escalation. Start with explainable rules, then compare routes using cost per successful task.
  5. Add reuse in risk order. Exact response cache first, provider prefix cache second, semantic cache only after false-hit review.
  6. Review the expensive tail weekly. The top one percent of runs often reveals loops, retrieval failure, broken tools, and missing stop conditions.

AI-engineer review

How to review cost optimizations written by a model

Check the economics

  • Does the code measure success, or only tokens?
  • Does routing include the cost of the router?
  • Does escalation happen selectively?
  • Are cache hits and misses observable?
  • Are batch savings worth the added latency?
  • Can one runaway agent exceed the budget?

Check correctness

  • Can trimmed context remove a critical constraint?
  • Can the memory summary drift from source events?
  • Does structured output still need semantic validation?
  • Does the cache key include permissions and freshness?
  • Can semantic similarity cross customer boundaries?
  • Does the stop condition return an auditable partial state?
The review question

Did this change reduce the cost of successful work, or did it merely make one API call look cheaper?
