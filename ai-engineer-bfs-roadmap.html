The progression

Build capability in the order that systems expose it.

The roadmap begins with software engineering because every AI product still has APIs, data contracts, tests, state, users, failures, and operating costs. Models add a probabilistic layer; they do not remove the engineering underneath.

Each stage adds a new kind of judgment. The goal is not to finish a technology checklist. The goal is to know what to build, why it should work, how to prove that it works, and where it will fail.

The ten stages

What to learn, build, and prove at every level

Move forward when you can demonstrate the stage in a working artifact. Familiarity is not the same as capability.

01

Foundations and engineering

Learn to build dependable software before adding probabilistic behavior.

  • Use Python and Git fluently enough to navigate, change, test, and review a real codebase.
  • Understand HTTP, JSON, authentication, status codes, and API contracts.
  • Write SQL and focused tests that make data and behavior verifiable.
02

ML and LLM fundamentals

Understand what the model receives, predicts, and cannot guarantee.

  • Learn tokens, context windows, embeddings, sampling, and basic evaluation.
  • Design prompts around clear instructions, context, examples, and output schemas.
  • Use model APIs directly before hiding behavior behind a framework.
03

Build small AI apps

Turn model calls into an observable user workflow.

  • Build typed APIs and use asynchronous calls where latency or concurrency matters.
  • Stream results with SSE and design the interface for partial, delayed, or failed output.
  • Add retries, timeouts, validation, and tests instead of treating the model call as a demo.
04

Data, search, and RAG

Ground generation in evidence the system can retrieve and cite.

  • Chunk and embed content according to document structure and user questions.
  • Combine lexical and semantic retrieval, then rerank when precision matters.
  • Evaluate retrieval separately from generation so the wrong layer gets fixed.
05

Tools and integrations

Connect models to data and actions through explicit contracts.

  • Implement tool calling and understand MCP tools, resources, and prompts.
  • Add authentication, input schemas, output schemas, permissions, and auditability.
  • Separate working memory, durable state, and external systems of record.
06

Agentic systems

Use agents when the problem requires stateful decisions across steps.

  • Model workflows as state graphs with explicit transitions and stopping conditions.
  • Persist durable checkpoints so runs can recover after timeouts and failures.
  • Put humans into the loop for ambiguity, high impact, and irreversible actions.
07

Full-stack AI products

Make the intelligence useful inside a complete product experience.

  • Connect frontend, API, model, and data flows without hiding failure states.
  • Add authentication, persistence, permissions, billing, and user-level isolation as needed.
  • Design feedback loops that capture corrections and improve future evaluations.
08

Evaluation, safety, and observability

Replace intuition with repeatable evidence.

  • Build representative datasets and combine deterministic checks with model-based graders.
  • Trace prompts, retrieval, tools, latency, cost, and state transitions end to end.
  • Run safety reviews for data exposure, prompt injection, permissions, and harmful actions.
09

Scale, secure, and deploy

Operate the product under real traffic, budgets, and security constraints.

  • Use containers and CI/CD to make builds reproducible and releases reversible.
  • Apply least-privilege IAM and keep secrets out of prompts, code, logs, and traces.
  • Use queues, caching, rate limits, and concurrency controls to protect dependent systems.
10

Lead, learn, and create impact

Connect technical choices to product outcomes and team capability.

  • Use system and product design to make tradeoffs explicit across quality, speed, cost, and risk.
  • Read research critically, write clear docs, and record decisions through ADRs and RFCs.
  • Lead reviews, mentor engineers, and make the system easier for the next person to understand.

The tool layer

Technologies to explore after the capability is clear

Tools change quickly. The underlying job changes more slowly. Pick from the ecosystem only after you can explain the responsibility it serves.

Build and intelligence

  • PyTorch and Transformers for model understanding and experimentation
  • Pydantic and FastAPI for typed contracts and APIs
  • pgvector or OpenSearch for retrieval infrastructure
  • MCP for portable tools, resources, and prompts
  • LangGraph or durable workflow engines for stateful execution
  • Next.js and Postgres for complete product delivery

Evidence and operations

  • LangSmith or OpenTelemetry for traces and operational evidence
  • OWASP guidance for application and LLM security review
  • Docker and GitHub Actions for reproducible delivery
  • Terraform for reviewable infrastructure changes
  • C4 diagrams for system communication
  • ADRs and RFCs for durable engineering decisions

Learning loop

Use projects as evidence, not tutorials as progress

Build one thin slice

Choose one real user workflow and complete it end to end. A narrow deployed system teaches more than several disconnected notebooks.

Break it deliberately

Test empty retrieval, malformed tool output, rate limits, stale state, model refusal, timeouts, and partial failures. Reliability grows through specific evidence.

Explain the tradeoff

For every framework or model choice, state what it improves, what it costs, and what would make you replace it. That is engineering judgment.

The AI-engineer test

Can you explain why every layer exists, verify model-generated code against its contracts, and debug the system when the happy path breaks? The roadmap is complete only when the answer is yes.

Portfolio proof

What a strong project should demonstrate

The artifact

  • A real user problem and measurable success condition
  • A working UI and API with authentication where appropriate
  • Versioned prompts, schemas, tests, and evaluation data
  • Observable retrieval, model, tool, and workflow behavior
  • A deployed system with a clear operating model

The explanation

  • Why AI is necessary for the workflow
  • How quality is measured beyond a polished demo
  • Where deterministic software replaces model judgment
  • How safety, cost, latency, and privacy shaped the design
  • What production evidence would trigger the next change
