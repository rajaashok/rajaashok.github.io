Claude Agent SDK: Put the Agent Loop Inside Your App

The useful mental model is simple: the model reasons, the SDK executes, and your application enforces the boundary conditions.

The core idea

The SDK is the Claude Code agent loop as a library

Claude Code already knows how to read files, edit code, run commands, search, ask for permissions, call tools, and keep working until the task is complete. The Claude Agent SDK gives you that harness from Python or TypeScript.

That means you do not write the low-level loop yourself. Your application sends a prompt and configuration. The SDK streams messages while the agent reasons, requests tool calls, executes approved tools, receives tool results, and eventually emits a final result or a controlled stop condition.

The mistake is treating the SDK like a normal chat completion wrapper. A chat wrapper asks for an answer. An agent SDK gives the model a controlled operating environment. That is why the design work is less about clever prompt text and more about boundaries: which tools exist, which tools auto-approve, which actions need review, what gets logged, and when the run must stop.

The loop

Prompt → reason → act → result → new context

The top row of the diagram is the feedback loop. Once you see that loop, the SDK API makes much more sense.

01

Prompt

Your application starts with a task, not a chat message alone.

The prompt should include the goal, constraints, and what a good finish looks like. For agentic work, the finish condition matters as much as the task description.

02

Claude reasons

The model decides what information or action is needed next.

It may answer directly, inspect files, run a command, call an MCP tool, or ask for permission. You should expect multi-step behavior, not one request and one response.

03

Tools act

The SDK and Claude Code runtime execute approved actions.

Built-in tools can read, edit, search, and run shell commands. Custom tools and MCP servers connect the loop to your actual application systems.

04

Result becomes context

Tool output comes back into the conversation.

The next model decision sees what happened. That is why tool outputs must be small, structured, and truthful. Messy tool results become messy agent reasoning.

The useful split: Claude chooses the next move. The SDK executes the move. Your application decides what moves are allowed.

Start

Install the SDK, then decide how much loop control you need

For Python, the package is claude-agent-sdk and requires Python 3.10 or later. The current Python package bundles the Claude Code CLI, so a separate Claude Code install is not required for the default path.

That install step is the easy part. The real first decision is which interface your product needs:

Use query() for one-off tasks

query() is an async stream. Give it a prompt and optional configuration, then process messages until the run completes. It is the cleanest path for short, fresh-session jobs: fix this file, summarize this folder, generate this report, inspect this failing test.

Use ClaudeSDKClient for interactive sessions

The client is for bidirectional, longer-running workflows: multi-turn control, streaming input, custom in-process tools, hooks as Python functions, session control, and applications that need to interrupt, resume, or steer the run.

A minimal working loop

The code shape that matters

Most SDK examples are short, but production code needs a little more structure: options, allowed tools, budgets, result handling, and logging.

pythonquery() with explicit controls
import asyncio
from claude_agent_sdk import query, ClaudeAgentOptions, ResultMessage

PROMPT = """
Fix the failing auth tests.
Verify the changed behavior with the existing test suite.
Do not modify unrelated files.
"""

async def run_agent():
    options = ClaudeAgentOptions(
        cwd="/path/to/project",
        allowed_tools=["Read", "Edit", "Bash", "Grep"],
        permission_mode="acceptEdits",
        max_turns=30,
        max_budget_usd=5.00,
    )

    async for message in query(prompt=PROMPT, options=options):
        if isinstance(message, ResultMessage):
            log_result(
                status=message.subtype,
                session_id=message.session_id,
                usage=message.usage,
                cost=message.total_cost_usd,
            )

            if message.subtype == "success":
                return message.result

            raise RuntimeError(f"Agent stopped: {message.subtype}")

asyncio.run(run_agent())

Read that code like an engineer, not like a demo. The important part is not that Claude can edit code. The important part is that the application has scoped where it can work, what it may auto-use, how long it can run, how much it can spend, and what evidence gets recorded at the end.

Options

The option object is your control surface

When a model writes SDK code, inspect the options first. Most production bugs hide in missing boundaries, not in the prompt string.

OptionWhat it controlsWhat to verify
cwdThe working directory for file and shell operations.Make sure the agent is not operating from a parent folder that exposes too much.
system_promptThe durable behavior instruction for this run.Keep it specific to the workflow. Avoid generic personality soup.
allowed_toolsTools that can be auto-approved.Remember: this is an auto-approval list, not the full availability boundary. Use disallow/blocking controls where needed.
permission_modeHow permission prompts are handled.Choose the narrowest mode that still lets the workflow complete.
mcp_serversExternal or in-process tool servers.Check credentials, data scope, timeout behavior, and whether tool names are allowlisted correctly.
hooksApp-side event handlers before or after tool activity.Use hooks for deterministic checks: block risky commands, audit output, save checkpoints.
max_turnsThe maximum number of agent round trips.Set a cap so a confused agent cannot loop forever.
max_budget_usdThe spend ceiling for the run.Use budgets on unattended or batch workflows.

Control the agent

Tools define capability. Permissions define authority. Hooks define policy.

Those three ideas are easy to blur together. Keep them separate and agent code becomes much easier to review.

01

Tools

What actions Claude can ask to perform.

Built-ins include file reads, edits, shell commands, search, and similar coding-agent actions. MCP and custom tools expose your application-specific capabilities.

02

Permissions

Which requested actions can proceed.

Permissions are where you decide whether an action is auto-approved, requires confirmation, or must be blocked. Auto-approval should be earned, not assumed.

03

Hooks

Deterministic application code around the loop.

Hooks can validate a tool request before it runs, audit outputs after it runs, or save state when the agent stops. They are not model judgment. They are your code.

04

Limits

Caps on turns, cost, runtime, and working scope.

Limits are the difference between a helpful autonomous worker and an unbounded background process. Every product workflow should have explicit stop conditions.

Hooks

Use hooks for the rules you cannot leave to vibes

A good prompt can ask the model to avoid dangerous commands. A hook can actually block one. That is the safety difference.

pythonPreToolUse guardrail
from claude_agent_sdk import ClaudeAgentOptions, ClaudeSDKClient, HookMatcher

async def block_risky_bash(input_data, tool_use_id, context):
    if input_data["tool_name"] != "Bash":
        return {}

    command = input_data["tool_input"].get("command", "")
    risky = ["rm -rf", "git reset --hard", "curl | bash"]

    if any(pattern in command for pattern in risky):
        return {
            "hookSpecificOutput": {
                "hookEventName": "PreToolUse",
                "permissionDecision": "deny",
                "permissionDecisionReason": "Blocked risky shell command",
            }
        }

    return {}

options = ClaudeAgentOptions(
    allowed_tools=["Read", "Edit", "Bash"],
    hooks={
        "PreToolUse": [
            HookMatcher(matcher="Bash", hooks=[block_risky_bash])
        ]
    },
)

Notice the pattern: the model can still propose a Bash command, but the application owns the final enforcement. This is how agent products become reviewable. You can test the hook with known-bad commands, inspect the denial reason, and prove the boundary works.

Extend

Custom tools are where your product enters the loop

Reading files and running shell commands are useful for coding tasks. Product agents need business actions: fetch an order, look up a customer, create a ticket, schedule a job, update a record, run a calculation.

The SDK supports MCP servers, and the Python client can define custom tools as in-process SDK MCP servers. That keeps simple app-owned tools in the same Python process instead of launching a separate server just to expose one function.

pythonin-process custom tool
from claude_agent_sdk import (
    tool,
    create_sdk_mcp_server,
    ClaudeAgentOptions,
    ClaudeSDKClient,
)

@tool("get_order_status", "Look up an order by id", {"order_id": str})
async def get_order_status(args):
    order = await orders_db.fetch(args["order_id"])
    return {
        "content": [
            {"type": "text", "text": f"status={order.status}; eta={order.eta}"}
        ]
    }

app_tools = create_sdk_mcp_server(
    name="app",
    version="1.0.0",
    tools=[get_order_status],
)

options = ClaudeAgentOptions(
    mcp_servers={"app": app_tools},
    allowed_tools=["mcp__app__get_order_status"],
)

async with ClaudeSDKClient(options=options) as client:
    await client.query("Check order A123 and explain the delay.")
    async for message in client.receive_response():
        print(message)

Design custom tools like APIs you expect a stranger to call correctly: narrow names, typed inputs, explicit outputs, stable errors, short responses, idempotency when possible, and no hidden side effects unless the tool name makes the side effect obvious.

Capture the result

The final answer is not the only output

A serious agent run produces operational evidence. Capture it. You will need it for debugging, billing, analytics, evaluation, support, and compliance.

Record every run

  • Prompt or task identifier.
  • Session ID or run ID.
  • Status or stop subtype.
  • Model, options, and tool configuration version.
  • Token usage, cost, and duration.
  • Tool calls and permission decisions.
  • Final result or failure reason.

Watch for bad endings

  • Turn limit hit before a result.
  • Budget limit hit mid-task.
  • Tool denied repeatedly.
  • Unstructured tool output confused the next step.
  • Agent changed unrelated files.
  • Final answer claims verification that never ran.
  • Session is not resumable or auditable.

Ship safely

The production checklist is boring on purpose

A demo agent optimizes for delight. A production agent optimizes for bounded autonomy.

01

Least privilege

Give the agent only the tools and data needed for the workflow.

  • Prefer read-only tools until writes are necessary.
  • Separate high-impact write tools from low-risk lookup tools.
  • Use app-side authorization, not just prompt instructions.
02

Cap cost and turns

Autonomy needs hard limits.

  • Set turn caps for every unattended workflow.
  • Set budget caps for batch or background jobs.
  • Alert when retries or denials spike.
03

Verify outputs

The agent should prove work, not just report confidence.

  • Run tests for code changes.
  • Use structured outputs where downstream code depends on fields.
  • Check files, database writes, or tickets actually changed as expected.
04

Log every run

If you cannot reconstruct it, you cannot debug it.

  • Trace model messages, tool calls, hook decisions, and final status.
  • Scrub secrets and sensitive payloads before storage.
  • Keep enough evidence to reproduce failures.
The AI-engineer review question

When a model writes Claude Agent SDK code for you, do not stop at "it imports correctly." Check whether it scoped tools, set limits, handled non-success results, logged usage, validated risky tool calls, and made verification observable.

Decision tree

When should you use the Agent SDK?

Use the Client SDK

When you want direct Claude API calls and you are comfortable managing prompts, tool calls, tool results, conversation history, and loop termination yourself.

Use the Agent SDK

When you want the Claude Code-style loop inside your own process, with built-in tool execution, context management, permissions, hooks, MCP, and sessions.

Use Managed Agents

When you want Anthropic-managed infrastructure and session storage instead of running the agent loop and environment in your own application process.
